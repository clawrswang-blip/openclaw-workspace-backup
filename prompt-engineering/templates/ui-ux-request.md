# ui-ux-request — UI/UX 设计需求

## 触发条件
- 你需要给设计师一个清晰的 Brief
- 目标：减少返工，一次讲清楚
- 关联：ui-ux-pro-max skill

## 输入变量

| 变量 | 说明 | 示例 |
|------|------|------|
| project_name | 项目名称 | Sungiven App 会员中心 |
| project_type | 项目类型 | App / Web / 小程序 |
| user_persona | 用户画像 | 25-40岁温哥华家庭主妇 |
| primary_goal | 核心目标 | 提升会员续费率 |
| key_flows | 关键流程 | 注册→会员等级→续费 |
| pain_points | 痛点 | 用户找不到续费入口 |
| design_constraints | 设计约束 | 品牌色、已有组件库 |
| reference_links | 参考链接（可选） | App A、B |

## 核心 Prompt

===新增===
【质量锚点：Top 3% 标准】
- 不是「够用」，是「让人想收藏」
- 不是「说明白了」，是「看完觉得值」
- 不是「通用答案」，是「这个细分领域的深层洞察」
- 不是「表面总结」，是「别人没看到的维度」
- 所有数据必须有来源，所有观点必须有依据
===新增===

你给的设计需求文档，不是"我要一个界面"，是"让设计师知道为什么要这样设计"。

---

**项目：** {project_name}
**类型：** {project_type}
**用户画像：** {user_persona}
**核心目标：** {primary_goal}
**关键流程：** {key_flows}
**痛点：** {pain_points}
**设计约束：** {design_constraints}
**参考（可选）：** {reference_links}

---

**第一步：毒辣之眼 — 设计需求文档最大的问题是什么？**

在动笔之前，先问：
- 设计师最大的困惑是什么？（"为什么要这样？"）
- 最大的返工原因是什么？（"没说清楚"还是"没想清楚"？）
- 你给的"参考链接"是"抄这个"还是"参考这个风格"？（这个要明确）

**第二步：需求分层**

```
[业务层]
  - {primary_goal}是什么意思？
  - 成功的衡量标准是什么？
  - 商业约束是什么？

[用户层]
  - {user_persona}是谁？
  - 他们的{pain_points}是什么？
  - 他们期望的体验是什么？

[技术层]
  - {design_constraints}
  - 平台限制
  - 性能要求
```

**第三步：关键流程描述**

{key_flows} 的每个步骤：
- 页面/界面名称
- 核心功能
- 用户目标
- 预期行为

**第四步：输出结构**

```
[项目概览]
  {project_name} / {project_type}

[核心目标]
  {primary_goal}
  成功衡量标准

[用户画像]
  {user_persona}
  核心需求
  使用场景

[关键流程]
  流程1
  流程2

[功能需求]
  [优先级：P0/P1/P2]

[设计约束]
  {design_constraints}

[参考风格]（如有）
  {reference_links}

[禁止项]
  [设计师不能做什么]

[交付物]
  [低保真/高保真/交互稿]
```

**第五步：自检清单**

- [ ] 设计师知道"为什么要这样设计"吗？
- [ ] 核心目标（{primary_goal}）明确了吗？
- [ ] 痛点（{pain_points}）描述清楚了吗？
- [ ] 关键流程（{key_flows}）有优先级吗？
- [ ] 交付物标准清楚吗？

===新增===
【去AI化检查】
□ 是否有具体的个人观察/数据点，而不是泛泛而谈？
□ 是否有不完美的、真实的细节？（AI总爱给完美答案）
□ 句式是否有变化，还是全是「首先...其次...最后...」？
□ 是否有反直觉的洞见，而不是安全的主流观点？
□ 是否避免了「不仅...而且...」「随着...」这类AI最爱用的句式？
□ 是否有情绪温度，还是全程冷静得像产品说明书？
□ 是否有具体的场景/案例，而不是笼统的「比如说...」？
===新增===

===新增===
【UI/UX设计类·工程思维强化】
□ 是否考虑了可扩展性和维护性？
□ 是否说明了技术选型的权衡？
□ 是否有错误处理和边界条件？
□ 注释是否说明了「为什么」而不是「是什么」？
===新增===

## 自我迭代检查（必做）

### 产出后自检（Before 输出）
- [ ] 我提供的洞见，读者在其他地方能轻易找到吗？（如果是，降级重写）
- [ ] 我的核心观点能用一句话概括吗？（如果不能，结构有问题）
- [ ] 最有争议的那个点，我有没有回避？（回避了=不够真诚）

### 产出后迭代（After 初稿）
问题 1：标题是否足够吸引，让 80% 的人想点进来？
问题 2：开头是否在 3 秒内抛出「钩子」？
问题 3：是否有至少一个「反直觉」的洞察？
问题 4：结尾是否给出了行动指引或思考延伸，而不是「以上就是...」？
问题 5：是否符合该平台的语言风格？（XHS vs 公众号 vs 知乎 vs 微博）
