# system-architecture — 系统架构设计

## 触发条件
- 你需要设计一个系统架构
- 目标：平衡"理想"和"现实"
- 场景：技术选型、架构评审、技术规划

## 输入变量

| 变量 | 说明 | 示例 |
|------|------|------|
| system_name | 系统名称 | Sungiven订单处理系统 |
| system_scope | 系统范围 | 订单→支付→履约 |
| scale_requirement | 规模要求 | 日均1000单，峰值5000 |
| availability | 可用性要求 | 99.9% |
| tech_preference | 技术偏好 | 已有技术栈 |
| budget_constraint | 预算约束 | C$5000/月 |
| team_capacity | 团队能力 | 2个全栈 |

## 核心 Prompt

===新增===
【质量锚点：Top 3% 标准】
- 不是「够用」，是「让人想收藏」
- 不是「说明白了」，是「看完觉得值」
- 不是「通用答案」，是「这个细分领域的深层洞察」
- 不是「表面总结」，是「别人没看到的维度」
- 所有数据必须有来源，所有观点必须有依据
===新增===

你设计的系统架构，不是"最先进"，是"最合适"。

**第一步：毒辣之眼 — 架构最大的失败是什么？**

在动笔之前，先问：
- 最常见的架构失败不是"技术选型错了"，是"over-engineering"——为什么？
- {team_capacity}能维护什么复杂度的系统？
- {budget_constraint}和{scale_requirement}矛盾吗？

**第二步：需求分析**

```
[功能需求]
  {system_scope}
  核心功能优先级

[非功能需求]
  {scale_requirement}
  {availability}
  性能要求
  安全要求

[约束]
  {tech_preference}
  {budget_constraint}
  {team_capacity}
```

**第三步：架构选项**

不是"只有一个正确答案"：

| 方案 | 优势 | 劣势 | 成本 | 适用场景 |
|------|------|------|------|---------|
| 方案A：单体 | 简单、快 | 扩展性差 | 低 | 小规模 |
| 方案B：微服务 | 扩展性好 | 复杂 | 高 | 大规模 |
| 方案C：混合 | 平衡 | 边界模糊 | 中 | 中等规模 |

**第四步：技术选型**

根据{tech_preference}和{team_capacity}：

```
[前端]
  [选择+理由]

[后端]
  [选择+理由]

[数据库]
  [选择+理由]

[基础设施]
  [选择+理由]

[监控/运维]
  [选择+理由]
```

**第五步：十年视角**

- 系统能支撑{scale_requirement}多久？
- 什么时候需要重构？
- 技术债务在哪里？

**第六步：输出结构**

```
[系统概览]
  {system_name}
  范围：{system_scope}

[需求分析]
  功能需求
  非功能需求
  约束

[架构方案]
  [推荐方案]
  [备选方案]

[技术选型]
  [每层详细]

[数据模型]
  [核心实体]

[部署架构]
  [基础设施]

[技术债务]
  [已知的妥协]

[演进路线]
  现在→6个月→1年
```

**第七步：自检清单**

- [ ] 架构复杂度匹配{team_capacity}吗？
- [ ] {budget_constraint}够吗？
- [ ] {scale_requirement}能满足吗？
- [ ] {availability}能实现吗？
- [ ] 技术债务识别了吗？

===新增===
【去AI化检查】
□ 是否有具体的个人观察/数据点，而不是泛泛而谈？
□ 是否有不完美的、真实的细节？（AI总爱给完美答案）
□ 句式是否有变化，还是全是「首先...其次...最后...」？
□ 是否有反直觉的洞见，而不是安全的主流观点？
□ 是否避免了「不仅...而且...」「随着...」这类AI最爱用的句式？
□ 是否有情绪温度，还是全程冷静得像产品说明书？
□ 是否有具体的场景/案例，而不是笼统的「比如说...」？
===新增===

===新增===

□ 是否有错误处理和边界条件？

## 自我迭代检查（必做）

### 产出后自检（Before 输出）
- [ ] 我提供的洞见，读者在其他地方能轻易找到吗？（如果是，降级重写）
- [ ] 我的核心观点能用一句话概括吗？（如果不能，结构有问题）
- [ ] 最有争议的那个点，我有没有回避？（回避了=不够真诚）

### 产出后迭代（After 初稿）
问题 1：标题是否足够吸引，让 80% 的人想点进来？
问题 2：开头是否在 3 秒内抛出「钩子」？
问题 3：是否有至少一个「反直觉」的洞察？
问题 4：结尾是否给出了行动指引或思考延伸，而不是「以上就是...」？
问题 5：是否符合该平台的语言风格？（XHS vs 公众号 vs 知乎 vs 微博）
