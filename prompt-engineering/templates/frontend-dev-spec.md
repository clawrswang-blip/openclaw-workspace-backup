# frontend-dev-spec — 前端开发规范

## 触发条件
- 你需要给前端开发一个清晰的技术规范
- 目标：减少沟通成本，一次写清楚
- 关联：frontend-design-3 skill

## 输入变量

| 变量 | 说明 | 示例 |
|------|------|------|
| project_name | 项目名称 | Sungiven 会员中心 |
| tech_stack | 技术栈 | React / Next.js / Tailwind |
| component_requirements | 组件要求 | 复用组件库 / 新建 |
| api_contract | API约定 | REST / GraphQL |
| performance_targets | 性能指标 | LCP < 2.5s / FID < 100ms |
| testing_requirements | 测试要求 | 单元测试 / E2E |

## 核心 Prompt

===新增===
【质量锚点：Top 3% 标准】
- 不是「够用」，是「让人想收藏」
- 不是「说明白了」，是「看完觉得值」
- 不是「通用答案」，是「这个细分领域的深层洞察」
- 不是「表面总结」，是「别人没看到的维度」
- 所有数据必须有来源，所有观点必须有依据
===新增===

你写的开发规范，不是"技术堆砌"，是"让开发者知道怎么实现、为什么这样实现"。

---

**项目：** {project_name}
**技术栈：** {tech_stack}
**组件要求：** {component_requirements}
**API约定：** {api_contract}
**性能指标：** {performance_targets}
**测试要求：** {testing_requirements}

---

**第一步：毒辣之眼 — 开发规范最大的问题是什么？**

在动笔之前，先问：
- 开发者最常问的问题是什么？（"这里为什么要这样？"）
- 最大的返工原因是什么？（"没提前说清楚"）
- 这个项目的技术债务风险在哪里？

**第二步：规范分层**

```
[架构层]
  - 技术栈选择理由
  - 项目结构
  - 模块边界

[编码层]
  - 命名规范
  - 代码风格
  - 注释要求

[组件层]
  - {component_requirements}
  - 组件设计原则

[API层]
  - {api_contract}
  - 错误处理规范

[性能层]
  - {performance_targets}
  - 优化策略
```

**第三步：代码质量标准**

- 一致性：风格统一
- 可维护性：代码可读
- 可测试性：{testing_requirements}
- 可访问性：A11Y 标准

**第四步：输出结构**

```
[项目概览]
  {project_name}
  技术栈：{tech_stack}

[架构说明]
  项目结构
  模块职责

[编码规范]
  命名
  代码风格
  Git规范

[组件规范]
  {component_requirements}
  组件设计原则

[API规范]
  {api_contract}
  请求/响应格式
  错误码

[性能要求]
  {performance_targets}
  优化策略

[测试要求]
  {testing_requirements}
  覆盖率要求
```

**第五步：自检清单**

- [ ] 开发者知道"怎么实现"了吗？
- [ ] 编码规范一致吗？
- [ ] API约定清晰吗？
- [ ] 性能目标可衡量吗？
- [ ] 测试要求明确吗？

===新增===
【去AI化检查】
□ 是否有具体的个人观察/数据点，而不是泛泛而谈？
□ 是否有不完美的、真实的细节？（AI总爱给完美答案）
□ 句式是否有变化，还是全是「首先...其次...最后...」？
□ 是否有反直觉的洞见，而不是安全的主流观点？
□ 是否避免了「不仅...而且...」「随着...」这类AI最爱用的句式？
□ 是否有情绪温度，还是全程冷静得像产品说明书？
□ 是否有具体的场景/案例，而不是笼统的「比如说...」？
===新增===

===新增===
【代码/技术类·工程思维强化】
□ 是否考虑了可扩展性和维护性？
□ 是否说明了技术选型的权衡？
□ 是否有错误处理和边界条件？
□ 注释是否说明了「为什么」而不是「是什么」？
===新增===

## 自我迭代检查（必做）

### 产出后自检（Before 输出）
- [ ] 我提供的洞见，读者在其他地方能轻易找到吗？（如果是，降级重写）
- [ ] 我的核心观点能用一句话概括吗？（如果不能，结构有问题）
- [ ] 最有争议的那个点，我有没有回避？（回避了=不够真诚）

### 产出后迭代（After 初稿）
问题 1：标题是否足够吸引，让 80% 的人想点进来？
问题 2：开头是否在 3 秒内抛出「钩子」？
问题 3：是否有至少一个「反直觉」的洞察？
问题 4：结尾是否给出了行动指引或思考延伸，而不是「以上就是...」？
问题 5：是否符合该平台的语言风格？（XHS vs 公众号 vs 知乎 vs 微博）
